Istio

Istio is an ingress controller and a service mesh implementation for Kubernetes.

  • https://istio.io/v1.12/docs/

What is a sidecar container?

Kubernetes defines the pod as the minimum unit that can be created and managed. A pod can execute one or more containers which are executed in a shared context (network, namespaces, and volumes).

Using a single container per pod is a common use case in which the application has all of the resources assigned to the pod.

Sometimes the application consists of multiple containers that are related to each other and need to share resources like network or storage. One container can serve the network requests, while the other container is caching the operations in a shared volume for faster response.

A sidecar container is a pattern designed to have an extra container inside a pod that handles a set of specialized operations in order to add features to an existing application without altering it.

The term resembles the sidecar attached to a motorcycle, in the same way the sidecar container is attached to the pod and connected to the application.

What is a Service Mesh?

The term service mesh refers to a technology used to manage communication between the microservices in an application.

The service mesh provides components such as traffic management, retry logic, metrics, and security.

Those components can be added to the application via a sidecar container that implements all extra features and serves as a proxy for the ingress and egress traffic of the pod.

What is Istio?

Istio is an implementation of a service mesh that injects a sidecar container on the target pods to handle the ingress and egress traffic and route the requests accordingly. <!-- Istio uses the envoy proxy containers to process the ingress and egress traffic of a pod. -‚Üí Istio creates a control plane that centralizes service mesh capabilities and a data plane that creates the structure of the mesh.

The data plane controls communications between services by injecting sidecar containers that capture traffic between microservices.

The Istio control plane

The control plane is the connection between all the envoy sidecar proxies and istiod. The control plane is responsible for injecting the envoy proxies as sidecar containers in the target pods, it also manages the configuration of all the envoy proxies. The control plane does not directly handle the network traffic in the mesh. Once the configuration is processed, it is distributed to every sidecar proxy so they never have to contact the istiod process for information.

The Istio data plane

The data plane is the connection between all the sidecar proxies that forms the service mesh. The ingress traffic directed to a pod is intercepted by the sidecar proxy, where the routing rules are evaluated and the request is directed to the application container. For egress requests, the sidecar intercepts the traffic, applies routing the rules, and sends the request to the desired service.
